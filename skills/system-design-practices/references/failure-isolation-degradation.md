# 相依失效隔離與優雅降級

核心：**一個相依掛掉，不該讓整個服務掛掉。** 事前決定每個相依失效時「退化成什麼」，
而不是讓錯誤級聯（cascade）。

## 各相依的失效模式與降級策略

| 相依 | 掛掉時預設行為（沒設計） | 應該降級成 |
|---|---|---|
| **Cache（Redis）** | 讀取失敗 → 整頁掛 | **仍可讀**：回源 DB（read-through）；必要時對熱資料**降載**（拉長 TTL / 部分功能關閉）。快取是加速層，不該是單點。 |
| **Queue（RabbitMQ）** | 寫入失敗 → 使用者操作失敗 | **寫入退化**：明確回可重試錯誤（503）讓前端 optimistic 保留，或短暫本地緩衝；不要當掉整個請求。 |
| **DB（PostgreSQL）** | 全掛 | **只讀 / 延遲**：讀走 cache、寫入進 queue 緩衝稍後落地；`/readyz` 反映真實狀態。 |
| **外部 API** | 阻塞 / 拖垮呼叫端 | **逾時 + 重試上限 + 熔斷 + fallback**；絕不無限等待。 |

> 這幾條正是把「相依掛掉仍能部分服務 + 自動降載」當成 NFR 明確寫進需求的價值。

## fail-closed vs fail-open——選對

- **fail-closed（失敗即拒絕）**：用在**安全 / 授權閘門**。例：跨租戶 / 營運者後台的 allowlist
  取不到就**全拒**（寧可不可用，不可外洩）。
- **fail-open（失敗即放行 / 降級可用）**：用在**可用性優先**的非安全路徑。例：cache 掛了就回源，
  不要因此擋掉讀取。
- 選錯的代價：安全路徑 fail-open = 資料外洩；可用性路徑 fail-closed = 沒事自己把服務關了。

## 隔離手段

- **逾時**：所有跨程序呼叫都要有逾時（DB、cache、queue、HTTP client），沒逾時 = 一個慢相依拖垮全部。
- **熔斷 / 重試上限**：連續失敗就快速失敗，別讓重試風暴加壓已經在喘的相依。
- **blast radius / 資源隔離**：把不同職責的元件分開（例：把 app 與 observability 拆成不同部署單位，
  一邊的操作不會誤刪另一邊——見 iterative-delivery operational-pitfalls 的 compose project 事故）。
- **控制面 vs 資料面**：高頻資料面（讀 / 寫互動）與低頻控制面（後台管理）分開設計限制與失效行為。

## 驗證降級真的有效

- `/readyz`（或等價健康檢查）要**如實反映**各相依狀態（`postgres down` / `redis down`），
  當作部署與告警的判斷入口。
- 定期（或用故障注入）驗證「拔掉某相依，服務是否如設計般降級」，而不是假設它會。
