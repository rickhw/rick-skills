# 資源安全：凡事有界

**沒有上限的東西遲早失控。** 設計時對每一種資源設界，並把飽和訊號納入監控。

## 為每一種資源設上界

| 資源 | 沒上界的後果 | 設界方式 |
|---|---|---|
| **重試 / 迴圈** | poison message 無限 requeue → CPU 空轉數天 | 有界重試 + DLQ（見 async-messaging-reliability） |
| **Log 量** | 每則訊息 / 請求狂寫 → 磁碟塞滿 + page cache 膨脹 + log 收集器燒 CPU | 正常路徑 log 要精簡；錯誤別在迴圈裡重複刷；容器 log 設 `max-size` / `max-file` 輪替 |
| **Queue 深度** | 生產 > 消費 → 佇列無限長 → 記憶體 / 磁碟爆 | 監控 lag；限流或擴消費者；必要時 TTL / 上限 |
| **連線數** | 連線洩漏 / 無上限 → 打爆 DB / 檔案描述子 | 連線池上限；每個請求逾時；`defer close` |
| **記憶體** | 無界快取 / 累積 → OOM | 快取設容量 + 淘汰；串流處理大資料 |
| **磁碟** | image / build cache / log / 備份累積 → 100% → DB crash | 部署後清舊 image、定期 prune、備份輪替、log 輪替、**磁碟用量告警** |

> 真實事故：docker build cache（16G）+ 未使用 image（6G）把 `/` 塞到 100% → postgres
> `No space left on device` crash → 連鎖故障。教訓：**容器化服務的磁碟是共享且會被 image / log 吃掉的資源，要主動管理。**

## 資料生命週期

- 每種資料都要有**保留策略**（永久 / N 天 / 軟刪後 N 天清）。
- 軟刪 / 隱藏的資料要有清理排程，別無限累積。
- 暫存 / 測試資料**當場清乾淨**（殘留常變成故障觸發源，如指向不存在使用者的訊息）。

## 限制口徑要一致

- 長度 / 配額 / rate limit 在**所有檢查點用同一種計法**：應用層與 DB CHECK 一致（字元就都用
  字元 rune，不要一邊 byte 一邊 char——CJK 會被誤擋）。
- 限制值集中定義（常數 / 設定），不要散落各處各自為政。

## 可觀測性作為設計輸入

設計服務時就決定「哪些飽和 / 健康訊號要有指標 + 告警」，而不是等出事才被動從趨勢圖發現：

- **飽和**：CPU / 記憶體 / 磁碟 %、連線池使用率。
- **佇列**：DLQ 深度、訊息重投率、queue lag、單一 queue processed-per-sec 異常。
- **請求**：RPS、5xx rate、latency p99。
- **相依**：DB / cache / broker 連線狀態、健康檢查結果。
- 關鍵閾值（如磁碟 > 85%、DLQ > 0 持續、重投率飆高）要**主動告警**。
