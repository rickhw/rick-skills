# 識別碼（Identifier / ID）設計

API 路徑中的 `{id}` 不是隨手就用自動遞增或 UUID 的小事。識別碼設計需要**刻意選擇**，
依資源的範圍、壽命、數量規模與使用者互動方式而定，而不是預設用最方便的選項。

## 核心概念

- **目的**：識別碼用來唯一地定位系統中的某個資源或物件（如身分證號、產品碼、DNS 網域、
  IP、IMEI）。
- **範圍（Scope）**：每個識別碼都有其適用範圍，範圍決定了唯一性的保證。身分證號在國境內
  唯一；全域 UUID 跨所有系統唯一；EC2 Instance ID 跨 region 與帳號維持全域唯一。
  **設計時先問：這個 ID 要在多大的範圍內唯一？**

## 兩個設計維度

### 維度一：可數性（Countability）

- **可數 ID**：數量有限（約 ≤1,000）。例如 AWS region 代碼、台灣郵遞區號、HTTP 狀態碼。
  可被列舉、明確列出。
- **不可數 ID**：數量龐大。例如 IMEI、MAC、網頁數。無法實際列舉，必須以演算法產生。

> 類比 DevOps 的「Cattle vs. Pets」：可數 ID 像有名字的寵物；不可數 ID 像成群的牲畜。

### 維度二：可理解性（Understandability）

結合兩維度可得一張對照表：

| 類別 | 可數 | 不可數 |
|---|---|---|
| **可讀** | AWS region 代碼（`ap-northeast-1`） | Docker 隨機名稱 |
| **部分可理解** | HTTP 狀態碼、台灣身分證號 | UUID、帶前綴的 AWS resource ID |
| **完全不透明** | 罕用（敏感系統） | MySQL 自動遞增 |

- **A1（可數＋可讀）**：用列舉。如 AWS region、執行個體類型（`t2.small`、`c5.large`）、
  遵循命名慣例的安全群組名稱。
- **A2（可數＋部分可理解）**：系統錯誤碼（`E10-03`）、身分證號格式、HTTP 狀態碼，都遵循
  可辨識的模式。
- **A3（可數＋不透明）**：罕用；保留給需對外揭露的敏感資訊。
- **B2（不可數＋部分可理解）**：「懶工程師的設計」。UUID 有公認結構；AWS resource ID 用
  前綴（`i-`、`vpc-`、`ebs-`）標示類別；JIRA 議題號＝專案碼＋序號。
- **B3（不可數＋不透明）**：只追求唯一性。MySQL 自動遞增屬此類——只確保不碰撞，不在意
  內容含義。

## 常見產生方式

| 方式 | 特性 | 適用 |
|---|---|---|
| **自動遞增（Auto-Increment）** | 資料庫原生、簡單高效；僅單機／單庫範圍唯一；可被猜測／枚舉 | 小型系統、唯一性需求不高 |
| **UUID** | 演算法產生、全域唯一（v1 時間、v4 隨機）；不可排序（v4） | 分散式系統、跨系統資料交換 |
| **TSID / ULID（時間排序）** | 時間有序、對資料庫索引友善 | 需要排序又要分散式唯一的鍵 |
| **Snowflake**（Twitter） | 時間戳＋機器 ID＋序號；有序、全域唯一、高吞吐 | 大量、分散式 ID 產生（DB 鍵、訊息佇列） |
| **KGS（Key Generation Service）** | 集中式伺服器統一發號、避免碰撞 | 需強唯一性與集中管理的企業系統 |

## 常見問題與對策

- **ID 碰撞**：通常源於選錯屬性當 ID（例如拿 hostname 當 ID）。對策：用子網域縮小範圍，
  或隨機命名＋屬性標籤（部門／專案）。
- **數量過於龐大、ID 太長難用**：
  - 用 QR／條碼編碼供掃描。
  - **縮小範圍（range narrowing）**：電商以兩階段縮減（門市 + 以電話／證號末碼反查）。
  - **反向索引**：用顧客屬性比對訂單，而非要使用者背誦冗長的訂單號。

## 縮小有效範圍（Scope 管理）

電商訂單號通常是全域的，但對消費者很難用。對策：
1. **地理縮範圍**：便利商店只放 50–200 種商品。
2. **反查（Reverse Lookup）**：用顧客電話或證號末碼定位訂單，大幅改善體驗。

## 對 RESTful API 設計的意涵

- **內部 vs. 外部識別碼**：面向使用者（UI、溝通）的 ID 與內部系統 ID，曝光與操作需求不同。
  別把可被枚舉的內部自動遞增鍵直接當成對外路徑 `{id}`——會洩漏規模、可被遍歷。對外偏好
  不可預測的 ID（UUID／ULID／帶前綴的隨機碼）。
- **可數資源用列舉**：在程式語言（enum）與 API 參數中列出，讓客戶端可直接查詢
  （例如狀態值、類型）。
- **不可數資源透過 API 查詢**：因為無法窮舉，需提供查詢端點讓使用者自行檢索。
- **可排序性**：若 ID 也作為分頁游標或排序鍵，選時間有序的方案（ULID／TSID／Snowflake）
  而非 UUIDv4。
- **帶前綴的 ID**：像 `i-0abc...`、`vpc-...` 這類前綴讓 ID 自帶類別語意，利於除錯與日誌辨識。

## 選擇識別碼的判準

1. 數量規模（可數 vs. 不可數）。
2. 消費者的理解需求（可讀／部分可理解／不透明）。
3. 單機 vs. 分散式情境。
4. 驗證與安全需求（能否被偽造／猜測；是否需要校驗碼）。
5. 效能與可排序性需求。

> 兩個作者點出仍待深化的面向：**驗證**（如何偵測唯一性被破壞或偽造）與
> **對外 vs. 對內操作**的差異。設計時值得一併考量。
