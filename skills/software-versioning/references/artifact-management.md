# 產出物管理（Artifact Management）

## 分支策略 vs 產出物管理 — 解決不同問題

| | 解決什麼 |
|---|---|
| **分支策略（Branch Strategy）** | **團隊協作**：多人如何高效協作、流程分工、部署管線複雜度 |
| **產出物管理（Artifact）** | **軟體交付標準化**：跨環境一致、可追溯、可重現的交付物 |

兩者互相獨立，不必為了配合對方而妥協。同理，**版本管理（SDLC 視角）**與
**版本控制系統（Git/SVN/P4 工具與分支操作）**也是兩件事——TPM/PM/PO 若掌握不住整個
SDLC，就掌握不住團隊運作的節奏。

## 兩層分類：dev vs rel

### Dev（開發版）
- 功能**尚未驗證**，以驗證功能為主，**只能團隊內部使用**。
- 可含 debug 能力、特殊後門、設定旗標、選擇性編譯。
- 可有**多個**（時間序概念）。
- 例：`RickLds-v1.0.0-dev-b20220406-1200.zip`（帶日期／時間戳）。

### Rel（發布版）
- 功能**已完成驗證**、經過認證程序，**可交付客戶**。
- 關閉或限制 debug（以巧妙方式保留必要存取）；最佳化、混淆／簽章。
- **每個發布週期恰好一個**。
- 例：`RickLds-v1.0.0-rel.zip`（以 md5/sha256/sha512 驗證）。

## Artifact ≠ Config（1 對多）

- **artifact 與 config 是 1 對多的關係**：同一個 artifact 配不同環境的 config。
- **不要把 config 編進 artifact**。這是 Single Codebase, Multiple Deployment 的前提，
  也確保各環境跑的是「同一份」可驗證的產出物。

## SemVer 套用於專案生命週期

- **0 → 1 階段（初創）**：從 `v0.1` 起，每個 sprint 遞增（`v0.2-dev`、`v0.3-dev`…）；
  首次對客戶發布時直接跳 `v1.0-rel`。
- **1 → 10 階段（成熟）**：標準遞增 `v1.1.0 → v1.2.0 → v1.3.0`（每次發布進 Minor）。
- 不變的鐵則：開發過程版本號不動，**發布程序完成後**才遞增。

## 「第一天就交付 Hello World」哲學

- 專案**第一天**就建立 artifact 交付基礎設施，而不是靠 IDE 的匯出功能。
- 類比單元測試：交付物從專案起點就該由 **CI server 產生**。
- 適用任何產出物類型：Web Service、前端、Library、Desktop、Console。
- 支撐 Four Pillars of Software Delivery（標準化交付凌駕於特定分支策略或工具選擇之上）。

## 常見錯誤

1. 開發過程中（發布程序尚未完成前）就遞增版本號。
2. 跨環境不區分 dev/rel 版本。
3. 把 config 編進 artifact。
4. 誤以為與敏捷方法衝突。
5. 以為只能用 SemVer（如 Chrome 單號版本一樣有效）。
