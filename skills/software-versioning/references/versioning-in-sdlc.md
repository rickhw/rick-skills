# 版本在 SDLC 中的流動

## 版本號是溝通的介面

版本在所有角色（PM、開發、QA、維運）之間建立共同基準，確保大家談論軟體狀態時口徑一致。
這是版本管理最核心的目的。

## 兩種策略：Version-First vs Version-Late

### Version-First（先給號，適合產品）
- 在階段**開始**時就宣告版本，使用 **SemVer**（`MAJOR.MINOR.PATCH`，如 3.2.0 → 3.3.0）。
- 由 **PM 宣布版本**，觸發首個 build。
- 適用：開源（Kubernetes、Dapr、.NET、Linux）、工具、函式庫、SDK、API。
- 需搭配 **LTS（長期支援）** 版本管理。

### Version-Late（後給號，適合專案）
- 在**發布／部署時**才決定版本。
- 適合快速迭代、無外部相依的專案（如對客網站）。
- 以 commit hook、GitOps、時間戳或 SemVer 計算**自動**產生。

## 不給號（Version-Late 用壞）的連鎖問題

- 「版本本身沒有啥意義，因為大家不會拿來溝通。」
- 跨角色沒人知道目前是哪一版。
- 沒有 artifact 可追溯性；各環境可能跑不同程式碼而**不被察覺**。
- QA 測試失去穩定基準；版本在測試中途漂移而無人發現。
- Provisioning／環境設定變得困難許多。

## 版本管理的策略性功能

1. **溝通標準化** — 所有利害關係人的統一基準。
2. **產品迭代節奏** — 定義週期性的發布節奏。
3. **功能清晰度** — 追蹤每一版有哪些能力。
4. **遷移規劃** — 能比較版本、制定升級策略。
5. **產出物完整性** — 跨環境部署同一份產出物（Single Codebase, Multiple Deployment）。

## Single 的唯一性

SemVer 維持跨多次部署的一致性：如同 Windows 11 各安裝都源自同一份來源 image，
確保「Single 的唯一性」——同一來源、可重現、跨環境一致。

## 忽視版本的三個根因

- 缺乏以客戶為中心的視角。
- 缺少正式的溝通介面標準（專案管理失職）。
- 過度偏重自動化而忽略流程紀律。

> 「工具都是為了輔助流程，主、次應該是標準程序為主。」自動化是次要，標準程序是主要。

## 與其他實務的關係

O11y 脈絡）與 `restful-api-design`（API 版本治理）一起運用。
